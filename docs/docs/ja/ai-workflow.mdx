       │
       ▼
GitHub Actions CI / デプロイ`}</code></pre>

  <Section
    eyebrow="// 02 · アクター"
    heading="誰が何をするか。"
  >
    <h3>人間</h3>
    <ul>
      <li>戦略、スコープ、フェーズ順序を設定する。</li>
      <li>作業を定義する Issue を起票または承認する。</li>
      <li><strong>PR をマージする。</strong> これは委譲しない。</li>
      <li>アカウント、支払い、外部向け人間の対話を処理する。</li>
      <li>認証情報を更新し、本番状態への <code>tofu apply</code> を承認する。</li>
    </ul>

    <h3>AI エージェント</h3>

    <table>
      <thead>
        <tr><th>エージェント</th><th>役割</th><th>状態</th></tr>
      </thead>
      <tbody>
        <tr><td>Claude Code</td><td>主要な実装者 + レビュアー</td><td>稼働中</td></tr>
        <tr><td>Dosu</td><td>Issue トリアージ &amp; Q&amp;A</td><td>計画中</td></tr>
      </tbody>
    </table>

    <p>
      <strong>Claude Code</strong> はこのリポジトリで二つの役割を担う:
    </p>
    <ul>
      <li><strong>実装者</strong> — Issue を読み、PR を作成し、スコープ外の作業を発見した場合はフォローアップ Issue を起票する。</li>
      <li><strong>レビュアー</strong> — 非ドラフトの PR すべてに対し、AGENTS.md / CLAUDE.md / リンクされた Issue に照らして diff を読み、単一の GitHub レビューを投稿する（<code>COMMENT</code> または <code>REQUEST_CHANGES</code>、<code>APPROVE</code> は行わない）。</li>
    </ul>
  </Section>

  <Section
    eyebrow="// 04 · ロードマップボード"
    heading="ボードが運用サーフェスだ。"
  >
    <p>
      すべての Issue と PR は、org レベルの Projects v2 <strong>Auto-add to project</strong>{' '}
      ワークフローにより自動的に{' '}
      <a href="https://github.com/orgs/aletheia-works/projects/1">vivarium ロードマップボード</a>{' '}
      に追加される。このドキュメントではなく、ボードが「現在進行中のもの」の運用サーフェスだ。
    </p>

    <p>
      ボードは PR と Issue の両方をホストし、それぞれ <code>is:pr</code> と{' '}
      <code>is:issue</code> でフィルタされた <strong>二つの並列ロードマップビュー</strong>{' '}
      を持つ。両ビューはマイルストーンスイムレーン（Phase 0〜6、
      <code>infra/github/milestones.tf</code> で定義）を共有するが、
      タイムライン軸には独自の Date フィールドペアを使用する:
    </p>

    <ul>
      <li><strong>PR</strong> — <code>Started At</code>（<code>opened</code> 時に設定）と <code>Merged At</code>（<code>closed &amp;&amp; merged</code> 時に設定）。マージなしでクローズされた PR は <code>Merged At</code> が空のまま。</li>
      <li><strong>Issue</strong> — <code>Opened At</code>（<code>opened</code> 時に設定）と <code>Closed At</code>（<code>closed &amp;&amp; state_reason == "completed"</code> 時に設定）。<code>not_planned</code> でクローズされた Issue は <code>Closed At</code> が空のまま——PR ルールに対応する。</li>
    </ul>
  </Section>

  <Section
    eyebrow="// 06 · ループのスケール"
    heading="プロジェクトが成長しても保持されるもの。"
  >
    <p>
      上記の形はスケールする: より多くのエージェント（CodeRabbit、Sweep AI など）を
      追加のレビュアーとして組み込めるが、「人間がマージする」という制約は変わらない。
      参加する人間が増えても（複数メンテナー化）、「AI が実装とレビューを担う」という
      分担は変わらない。新しいレイヤー（Layer 3、将来の Layer 4）は実装者のスコープを
      拡張することで追加され、サイクル自体は変わらない。
    </p>

    <Callout>
      重要なのは AI が労働力の乗数であることではない。AI が、中心にいる人間を
      燃え尽かせることなく、終わりのないプロジェクトを可能にするメカニズムであることだ。
    </Callout>
