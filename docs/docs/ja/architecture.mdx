    <p>
      レイヤーが存在するのは、再現問題の性質が異なるからであり、
      技術が面白いからではない。Python インタープリタさえあれば再現できるバグに、

    </p>
    <Callout>
      正解のレイヤーは存在しない。問題がレイヤーを選ぶ。
    </Callout>
    <p>
      三層の分割は、新しい再現ドメインが現れたときにすでにスロットが用意されているようにするためにある。
      もし用意がなければ、第四のレイヤー（または既存レイヤー内の新技法）が必要なサインだ。
    </p>
  </Section>

  <Section
    eyebrow="// 03 · LAYER 1 — 詳細"
    heading="WebAssembly、ページの中で。"
  >
    <h3>対象ユースケース</h3>
    <ul>
      <li>アルゴリズムバグ（ソート、パース、テキスト処理）。</li>
      <li>データ処理のリグレッション（pandas / polars / numpy のシェイプバグ）。</li>
      <li>データベースの動作バグ（sqlite のクエリ異常、JSON パスのエッジケース）。</li>
      <li>ファイルシステム、ネットワーク、プロセスモデルに触れないライブラリ内部ロジック。</li>
    </ul>

    <h3>起動時間の特性</h3>
    <ul>
      <li><strong>コールドロード</strong>（初回訪問、service-worker キャッシュなし）: 1〜10 秒。ランタイムのダウンロード（Pyodide wheel bundle ≈ 10 MB 圧縮）とモジュールのインスタンス化が支配的。</li>
      <li><strong>ウォームロード</strong>（キャッシュ済みランタイム）: 1 秒未満でレディ状態。</li>
      <li><strong>再現実行ごと</strong>: Layer 1 が適切な問題では、ミリ秒〜数秒。</li>
    </ul>

    <h3>既知の制限</h3>
    <ul>
      <li><strong>メモリ。</strong> Pyodide タブはピーク ~1 GB 超で性能が劣化する。4 GB が実質的な上限。本番規模のデータは Layer 2 へ。</li>
      <li><strong>システムコール。</strong> 本物の <code>fork</code>/<code>exec</code>、本物のソケットは使えない。WASI シムが一般的なケースをカバーするが全てではない。</li>
      <li><strong>非決定論的インターリーブバグ。</strong> デフォルトはシングルスレッド。レースコンディションは Layer 3 へ。</li>
      <li><strong>コンパイル言語のツールチェーン。</strong> Rust のバグをブラウザ内で実行するのは問題ないが、Rust コンパイラ自体のビルドは不可（ホストツールチェーン → Layer 2）。</li>
    </ul>

    <h3>候補ランタイム</h3>
    <table>
      <thead>
        <tr><th>言語</th><th>ランタイム</th><th>状態</th></tr>
      </thead>
      <tbody>
        <tr><td>Python</td><td><a href="https://pyodide.org">Pyodide</a></td><td>Phase 1 ターゲット</td></tr>
        <tr><td>SQLite</td><td><a href="https://sqlite.org/wasm/">sqlite-wasm</a></td><td>Pyodide と同時</td></tr>
        <tr><td>Rust</td><td><code>wasm32-wasi</code> / <code>wasm32-unknown-unknown</code></td><td>Phase 2</td></tr>
        <tr><td>JavaScript</td><td>ネイティブブラウザ</td><td>Phase 2</td></tr>
        <tr><td>Ruby</td><td><a href="https://github.com/ruby/ruby.wasm">Ruby.wasm</a></td><td>Phase 2</td></tr>
        <tr><td>PHP</td><td><a href="https://github.com/WordPress/wordpress-playground">php-wasm</a></td><td>Phase 2</td></tr>
      </tbody>
    </table>
  </Section>

  <Section
    eyebrow="// 04 · LAYER 2 — 詳細"
    heading="Docker / microVM、訪問者のマシン上で。"
  >
    <h3>対象ユースケース</h3>
    <ul>
      <li>プロジェクト全体の再現——バグが実際のパッケージインデックスに対してプロジェクトの依存関係リゾルバを必要とする。</li>
      <li>システムコール依存バグ——本物の <code>fork</code>、本物のソケット、本物のファイルロック、本物のシグナル。</li>
      <li>ツールチェーン固有バグ——特定の GCC、特定の glibc、特定のカーネル ABI の癖。</li>
      <li>プロセス間の相互作用がバグの原因であるマルチプロセス / マルチコンテナバグ。</li>
    </ul>

    <h3>デリバリーモデル: SaaS 実行ではなくカタログ</h3>
    <p>
      Layer 2 はホスト型サンドボックスサービスではなく、<strong>再現可能なレシピのカタログ</strong>だ。
      各ギャラリーエントリにはピン留めされた <code>Dockerfile</code>、Layer 1 と同じ{' '}
      <code>pass</code> / <code>fail</code> verdict 形式を出力する <code>repro.sh</code>、
      そして <code>ghcr.io/aletheia-works/vivarium-&lt;slug&gt;</code> に公開された
      ビルド済みイメージが含まれる。ページにはコピペ可能な <code>docker run …</code> 
      コマンドと CI の verdict スナップショットが表示され、訪問者自身の実行がライブ確認となる。
      該当する場合は「Open in Codespaces」バッジも提供する。
    </p>

    <h3>既知の制限</h3>
    <ul>
      <li><strong>ページ内実行なし。</strong> Layer 2 の再現はギャラリーページ内では実行されない——訪問者がローカルまたは Codespaces で実行する。</li>
      <li><strong>Docker が必要</strong>（または Codespaces アカウント）。どちらもない場合でも、レシピと verdict スナップショットは閲覧できる。</li>
      <li><strong>サンドボックスの信頼境界。</strong> 訪問者が自身の Docker で Layer 2 再現を実行するため、Vivarium は未検証コードのホストではない——信頼境界は訪問者自身のランタイムとなる。</li>
      <li><strong>コストは訪問者側に。</strong> イメージストレージ、帯域幅、CI 分数はすべてパブリックリポジトリ / パブリックイメージの無料枠内に収まる。</li>
    </ul>
  </Section>

  <Section
    eyebrow="// 05 · LAYER 3 — 詳細"
    heading="トレースを同梱し、どこでも replay。"
  >
    <h3>対象ユースケース</h3>
    <ul>
      <li>ハイゼンバグ: データレース、メモリ順序バグ、単純な再実行では再現しない use-after-free。</li>
      <li>長時間 replay シナリオ: 何時間ものキャプチャ実行を数分で replay し、タイムトラベルデバッグを行う。</li>
      <li>特定のメッセージインターリーブに依存する分散システムバグ。</li>
      <li>事後フォレンジック分析——新規実行から前に進むのではなく、録音済みトレースを後ろに戻る。</li>
    </ul>

    <h3>デリバリーモデル: トレース同梱カタログ</h3>
    <p>
      Layer 3 はホスト型 record-replay サービスではなく、
      <strong>事前録音トレースをパッケージした自己完結イメージのカタログ</strong>だ。
      キャプチャはレシピごとに一度、オフラインで実施する。
      メンテナーが使用可能な PMU を持つ Linux/x86_64 ホストで録音し、
      トレースをタグピン留めのリリースアセットとしてアップロードし、
      CI がビルド時にイメージに焼き込む。
      <code>rr replay</code> は PMU を必要としないため、
      訪問者と CI の replay はどちらもコモディティ Linux 上で動作する。
    </p>

    <h3>既知の制限</h3>
    <ul>
      <li><strong>再現ごとのコストが高い。</strong> Layer 3 は Layer 1 と Layer 2 が失敗を観測できない場合にのみ使用する——Layer 2 のアップグレードパスとしてではない。</li>
      <li><strong>ツール固有のカバレッジギャップ。</strong> <code>rr</code> はシングルプロセスの Linux x86_64 を対象とし、分散シミュレーションツールは分散システムを対象とする——互いの領域はカバーしない。</li>
      <li><strong>録音はメンテナーの責任、CI ではない。</strong> Layer 3 レシピの追加には、動作する PMU を持つ Linux/x86_64 ホストへのアクセスが必要。</li>
      <li><strong>ストレージ。</strong> トレースは大きい（レシピあたり数十 MB から一桁 GB）。</li>
      <li><strong>訪問者は Linux/x86_64 が必要。</strong> ARM Mac や Windows の訪問者は Codespaces または WSL2 + Docker にフォールバックする。</li>
    </ul>
  </Section>

  <Section
    eyebrow="// 06 · ルーティング"
    heading="再現リクエストのルーティング方法。"
  >
    <p>
      ルーティングは副次的な関心事ではなく、第一級の設計事項だ。現時点で機械的なルーターは存在しない——
      Layer 1 と Layer 2 のカタログはディレクトリレベルのスロットとして存在し、
      既存のレシピはすべて作成時に手動でレイヤーを割り当てている。
      選択ルールは意図的に保守的だ:
    </p>

    <NumberedList
      items={[
        {
          lead: 'Layer 1 がデフォルトで勝つ。',
          body: 'Layer 1 で再現を忠実に実行できるなら、Layer 1 が勝つ。初回描画が速く、ホスティングコストも安く、アカウント不要で動作する。',
        },
        {
          lead: 'Layer 2 が残りをカバーする。',
          body: 'Layer 1 が対応できないが Layer 2 が要件を満たすなら、Layer 2 が勝つ。本物の OS、本物のプロセス、起動レイテンシのコストと引き換えに。',
        },
        {
          lead: 'Layer 3 は他のレイヤーが観測できないバグだけ。',
          body: '並行性、非決定論、タイムトラベルが必要な場合。Layer 3 は Layer 2 のアップグレードではなく、異なるバグのための異なるプロダクトだ。',
        },
      ]}
    />

    <p>
      曖昧なケースは、証拠が出るまで最も安価なレイヤーにとどまる。
      より高価なレイヤーへの昇格には、勘ではなく具体的な失敗シグナルが必要だ。
    </p>
  </Section>

  <Section
    eyebrow="// 07 · インターフェース"
    heading="レイヤー横断で統一された単一の宣言形式。"
  >
    <p>
      再現はコンパクトなレイヤー非依存の仕様で自己宣言する。
      ブラウザタブで動作しても Firecracker microVM で動作しても、
      同じ宣言形式が使えることを意図している:
    </p>
    <ul>
      <li><strong>Identity</strong> — 再現対象のアップストリームリポジトリ + Issue / commit / バージョン。</li>
      <li><strong>Inputs</strong> — バグをトリガーするために必要な決定論的入力。</li>
      <li><strong>Expected failure signal</strong> — 「再現した」とはどういう状態か（終了コード、特定のエラーテキスト、構造化 verdict）。</li>
      <li><strong>Runtime hints</strong> — 最小要件（Python 3.12 以降、glibc ≥ 2.36、network: no、filesystem: read-only など）。</li>
      <li><strong>Preferred layer（任意）</strong> — 報告者が特定の理由を持つ場合にのみ設定。ルーターのデフォルト判断が正しいことが期待される。</li>
    </ul>
  </Section>
