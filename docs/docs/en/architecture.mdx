  <Section
    eyebrow="// 01 · GUIDING PRINCIPLE"
    heading="Technology serves the problem."
  >
    <p>
      The layers exist because reproduction problems differ in kind, not because
      the technologies are interesting. A bug that only needs a Python interpreter
      to reproduce should not pay the startup cost of a container. A bug that
      needs <code>fork()</code> must not be pretended into a WASM sandbox that
      cannot run it. A heisenbug that fails to reproduce one in ten runs cannot
      be solved by running it harder.
    </p>
    <Callout>
      No layer is the right answer. The problem chooses the layer.
    </Callout>
    <p>
      The three-layer split exists so that when a new reproduction domain shows
      up, we already have a slot for it — and if we do not, that is the signal
      that a fourth layer (or a new technique inside an existing layer) might be
      warranted.
    </p>
  </Section>

  <Section
    eyebrow="// 03 · LAYER 1 — IN DETAIL"
    heading="WebAssembly, in the page."
  >
    <h3>Target use cases</h3>
    <ul>
      <li>Algorithmic bugs (sorting, parsing, text processing).</li>
      <li>Data-processing regressions (pandas / polars / numpy shape bugs).</li>
      <li>Database behavioural bugs (sqlite query anomalies, JSON path edge cases).</li>
      <li>Library-internal logic where the repro does not touch the filesystem, network, or process model.</li>
    </ul>

    <h3>Startup-time characteristics</h3>
    <ul>
      <li><strong>Cold load</strong> (first visit, no service-worker cache): 1–10s, dominated by runtime download (Pyodide wheel bundle ≈ 10 MB compressed) and module instantiation.</li>
      <li><strong>Warm load</strong> (cached runtime): &lt;1s to ready state.</li>
      <li><strong>Per-reproduction execution</strong>: milliseconds to a few seconds, in the regime where Layer 1 is the right fit.</li>
    </ul>

    <h3>Known limits</h3>
    <ul>
      <li><strong>Memory.</strong> Pyodide tabs commonly degrade past ~1 GB peak; 4 GB is an effective hard ceiling. Production-scale data belongs in Layer 2.</li>
      <li><strong>System calls.</strong> No real <code>fork</code>/<code>exec</code>, no real sockets. WASI shims cover common cases but not all.</li>
      <li><strong>Deterministic-interleaving bugs.</strong> Single-threaded by default; race conditions belong in Layer 3.</li>
      <li><strong>Compiled-language toolchains.</strong> Running a Rust bug in-browser is fine; rebuilding the Rust compiler is not (host toolchain → Layer 2).</li>
    </ul>

    <h3>Candidate runtimes</h3>
    <table>

  <Section
    eyebrow="// 04 · LAYER 2 — IN DETAIL"
    heading="Docker / microVM, on the visitor’s machine."
  >
    <h3>Target use cases</h3>
    <ul>
      <li>Whole-project reproductions — the bug needs the project's actual dependency resolver against a real package index.</li>
      <li>System-call-dependent bugs — real <code>fork</code>, real sockets, real file locking, real signals.</li>
      <li>Toolchain-specific bugs — a particular GCC, a particular glibc, a particular kernel ABI quirk.</li>
      <li>Multi-process / multi-container bugs where the interaction <em>between</em> processes is the bug.</li>
    </ul>

    <h3>Delivery model: catalogue, not execution SaaS</h3>
    <p>
      Layer 2 is a <strong>catalogue of reproducible recipes</strong>, not a hosted-sandbox
      service. Each gallery entry ships a pinned <code>Dockerfile</code>, a{' '}
      <code>repro.sh</code> that emits the same <code>pass</code> /{' '}
      <code>fail</code> verdict shape Layer 1 uses, and a pre-built image
      published to <code>ghcr.io/aletheia-works/vivarium-&lt;slug&gt;</code>. The
      page surfaces a copy-pasteable <code>docker run …</code> invocation and a
      CI verdict snapshot; the visitor's own run is the live confirmation. An
      "Open in Codespaces" badge is offered where applicable, opening the same
      image in a one-click cloud devcontainer billed to the visitor's GitHub
      account.
    </p>

    <h3>Known limits</h3>
    <ul>
      <li><strong>No in-page run.</strong> Layer 2 reproductions never execute inside the gallery page — the visitor invokes them locally or in Codespaces.</li>
      <li><strong>Visitor needs Docker</strong> (or a Codespaces account). Without either, the recipe and verdict snapshot are still readable.</li>
      <li><strong>Sandboxing trust boundary.</strong> Visitors run Layer 2 reproductions on their own Docker, so Vivarium is not the host of the untrusted code — the trust boundary is the visitor's own runtime.</li>
      <li><strong>Cost on the visitor side</strong>, not Vivarium's. Image storage, bandwidth, and CI minutes are all in the public-repo / public-image free tiers.</li>
    </ul>
  </Section>

  <Section
    eyebrow="// 05 · LAYER 3 — IN DETAIL"
    heading="Trace-shipped, replayed anywhere."
  >
    <h3>Target use cases</h3>
    <ul>
      <li>Heisenbugs: data races, memory-ordering bugs, use-after-free where a naive rerun does not reproduce the failure.</li>
      <li>Long-replay scenarios: hours of captured execution replayed in minutes, with time-travel debugging.</li>
      <li>Distributed-system bugs where the failure depends on a specific message interleaving.</li>
      <li>Post-mortem forensic analysis — stepping backwards through a recorded trace rather than forwards from a fresh run.</li>
    </ul>

    <h3>Delivery model: trace-shipped catalogue</h3>
    <p>
      Layer 3 is a <strong>catalogue of pre-recorded traces</strong> packaged into
      self-contained images, not a hosted record-replay service. Capture is
      offline and one-time per recipe: the maintainer records on a Linux/x86_64
      host that exposes a usable PMU, uploads the trace as a tag-pinned release
      asset, and CI bakes it into the image at build time.{' '}
      <code>rr replay</code> does not need the PMU, so visitor and CI replays
      both work on commodity Linux.
    </p>

    <h3>Known limits</h3>
    <ul>
      <li><strong>Cost per reproduction is high.</strong> Layer 3 runs only when Layers 1 and 2 cannot observe the failure — not as a default upgrade path.</li>
      <li><strong>Tool-specific coverage gaps.</strong> <code>rr</code> targets single-process Linux x86_64; distributed simulation tools target distributed systems; neither covers the other.</li>
      <li><strong>Recording is a maintainer responsibility, not CI's.</strong> Adding a Layer 3 recipe requires access to a Linux/x86_64 host with a working PMU.</li>
      <li><strong>Storage.</strong> Traces are large (tens of MB to single-digit GB per recipe).</li>
      <li><strong>Visitor needs Linux/x86_64.</strong> ARM-Mac and Windows visitors fall back to Codespaces or WSL2 + Docker.</li>
    </ul>
  </Section>

  <Section
    eyebrow="// 06 · ROUTING"
    heading="How a reproduction request is routed."
  >
    <p>
      Routing is a first-class concern, not a side effect. No mechanical router
      exists yet — the Layer 1 and Layer 2 catalogues live as directory-level
      slots, and every existing recipe was hand-routed to its layer at
      recipe-creation time. The selection rules are intentionally conservative:
    </p>

    <NumberedList
      items={[
        {
          lead: 'Layer 1 wins by default.',
          body: 'If Layer 1 can run the reproduction faithfully, Layer 1 wins. Faster first paint, cheaper to host, runs without an account.',
        },
        {
          lead: 'Layer 2 covers the rest.',
          body: 'If Layer 1 cannot, but Layer 2 covers the requirements, Layer 2 wins. Real OS, real processes, at the cost of startup latency.',
        },
        {
          lead: 'Layer 3 only for bugs the others cannot observe.',
          body: 'Concurrency, non-determinism, time-travel needs. Layer 3 is not an upgrade on Layer 2 — it is a different product for different bugs.',
        },
      ]}
    />

    <p>
      Ambiguous cases stay in the cheapest feasible layer until evidence says
      otherwise. Promoting a reproduction to a more expensive layer requires a
      concrete failure signal, not a hunch.
    </p>
  </Section>

  <Section
    eyebrow="// 07 · INTERFACE"
    heading="A single declaration shape across layers."
  >
    <p>
      A reproduction declares itself via a small, layer-agnostic spec. The
      intent is that the same declaration shape works whether the reproduction
      runs in a browser tab or a Firecracker microVM:
    </p>
    <ul>
      <li><strong>Identity</strong> — upstream repo + issue / commit / version being reproduced.</li>
      <li><strong>Inputs</strong> — deterministic inputs required to trigger the bug.</li>
      <li><strong>Expected failure signal</strong> — what "reproduced" looks like (exit code, specific error text, structured verdict).</li>
      <li><strong>Runtime hints</strong> — minimum requirements (Python 3.12+, glibc ≥ 2.36, network: no, filesystem: read-only, etc.).</li>
      <li><strong>Preferred layer (optional)</strong> — only set when the reporter has a specific reason; the router's default decision is expected to be right.</li>
    </ul>
  </Section>
