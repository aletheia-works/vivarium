  <Section
    eyebrow="// 01 · THE LOOP"
    heading="A non-autonomous cycle, on purpose."
  >
    <p>
      Every step in the cycle is either a human decision (intent, merge), an AI
      contribution (implement, review), or automation (CI, labels, release).
      The cycle is intentionally non-autonomous at the merge point: human
      judgement closes the loop.
    </p>

    <pre><code>{`Human intent ──▶ Issue (GH) ──▶ Claude Code implements
                                       │
                                       ▼
Human merge ◀── Claude Code reviews ◀── PR (draft → ready)
       │
       ▼
GitHub Actions CI / deploy`}</code></pre>

    <Callout>
      Governing rules live in AGENTS.md; this page is the process layer on top
      of them.
    </Callout>
  </Section>

  <Section
    eyebrow="// 02 · ACTORS"
    heading="Who does what."
  >
    <h3>Human</h3>
    <ul>
      <li>Set strategy, scope, and phase ordering.</li>
      <li>File or approve Issues that define work.</li>
      <li><strong>Merge PRs.</strong> Never delegated.</li>
      <li>Handle accounts, payments, and external-facing human dialogue.</li>
      <li>Rotate credentials and approve <code>tofu apply</code> against production state.</li>
    </ul>

    <h3>AI agents</h3>

    <table>
      <thead>
        <tr><th>Agent</th><th>Role</th><th>Status</th></tr>
      </thead>
      <tbody>
        <tr><td>Claude Code</td><td>Primary implementer + reviewer</td><td>Active</td></tr>
        <tr><td>Dosu</td><td>Issue triage &amp; Q&amp;A</td><td>Planned</td></tr>
      </tbody>
    </table>

    <p>
      <strong>Claude Code</strong> plays two roles in this repo:
    </p>
    <ul>
      <li><strong>Implementer</strong> — reads Issues, drafts PRs, opens follow-up Issues when it discovers out-of-scope work.</li>
      <li><strong>Reviewer</strong> — on every non-draft PR, reads the diff against AGENTS.md / CLAUDE.md / the linked Issue and posts a single GitHub review (<code>COMMENT</code> or <code>REQUEST_CHANGES</code>, never <code>APPROVE</code>).</li>
    </ul>
  </Section>

  <Section
    eyebrow="// 03 · GUARDRAILS"
    heading="What AI agents must and must not do."
  >
    <NumberedList
      items={[
        {
          lead: 'Never merge or approve PRs.',
          body: 'Reviews land as COMMENT or REQUEST_CHANGES; merge is the human-only edge of the loop. The Claude review workflow has no merge permission.',
        },
        {
          lead: 'Never force-push to main once the first PR has landed.',
          body: 'Pre-PR rebases against the initial commit are allowed; post-PR force-pushes are not. Branch protection enforces this on the GitHub side.',
        },
        {
          lead: 'Never rotate or create production secrets.',
          body: 'PROJECTS_TOKEN and similar tokens are issued by the human, scoped to the smallest viable surface, and rotated on the human’s schedule.',
        },
        {
          lead: 'Never run tofu apply against production state without human review.',
          body: 'Plan output gets posted to the PR; the human reads the plan before the apply runs. The CI workflow pauses before apply.',
        },
        {
          lead: 'Pivot scope only with human confirmation.',
          body: 'When the agent discovers out-of-scope work mid-task, it opens a follow-up Issue rather than expanding the current PR. This keeps reviews bounded.',
        },
      ]}
    />
  </Section>

  <Section
    eyebrow="// 04 · ROADMAP BOARD"
    heading="The board is the operational surface."
  >
    <p>
      All Issues and PRs land on the{' '}
      <a href="https://github.com/orgs/aletheia-works/projects/1">vivarium roadmap board</a>{' '}
      automatically via the org-level Projects v2 <strong>Auto-add to project</strong>{' '}
      workflow. The board — not this document — is the operational surface for
      "what is in flight right now."
    </p>

    <p>
      The board hosts both PRs and issues, with <strong>two parallel Roadmap views</strong>{' '}
      filtered to <code>is:pr</code> and <code>is:issue</code> respectively.
      Both views share the milestone swimlanes (Phase 0–6, defined in{' '}
      <code>infra/github/milestones.tf</code>) but use their own pair of Date
      fields on the timeline axis:
    </p>

    <ul>
      <li><strong>PRs</strong> — <code>Started At</code> (set on <code>opened</code>) and <code>Merged At</code> (set on <code>closed &amp;&amp; merged</code>). Closed-without-merge PRs leave <code>Merged At</code> empty.</li>
      <li><strong>Issues</strong> — <code>Opened At</code> (set on <code>opened</code>) and <code>Closed At</code> (set on <code>closed &amp;&amp; state_reason == "completed"</code>). Issues closed as <code>not_planned</code> leave <code>Closed At</code> empty, mirroring the PR rule.</li>
    </ul>
  </Section>

  <Section
    eyebrow="// 06 · SCALING THE LOOP"
    heading="What the cycle preserves as the project grows."
  >
    <p>
      The shape above scales: more agents (CodeRabbit, Sweep AI, others) plug
      in as additional reviewers without changing the human-merges constraint.
      More humans can join (multiple maintainers) without changing the
      AI-implements-and-reviews split. New layers (Layer 3, future Layer 4) get
      added by extending the implementer's scope, not the cycle itself.
    </p>

    <Callout>
      The point isn't AI as a labour multiplier. The point is AI as the
      mechanism that lets a lifelong project happen without burning out the
      human at the centre.
    </Callout>
  </Section>
