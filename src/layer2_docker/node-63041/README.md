# Reproduction — Node.js `Intl.DateTimeFormat` drops month with `calendar: 'iso8601'`

> First **Layer 2 (Docker)** reproduction targeting an
> upstream-open Node.js bug. Conforms to the catalogue conventions
> in [`src/layer2_docker/README.md`](../README.md): pinned
> `Dockerfile` + `repro.sh` + a public pre-built image, with the
> visitor reproducing the bug locally via `docker run`.

## The behaviour

A `Intl.DateTimeFormat` instance configured with `dateStyle: 'full'`
must emit the full date — including the month name — regardless of
the choice of calendar. As of Node v24.13.0, supplying
`calendar: 'iso8601'` causes the formatter to silently elide the
month label from the output:

```js
const dtf = new Intl.DateTimeFormat('en-US', {
  dateStyle: 'full',
  timeStyle: 'long',
  timeZone: 'UTC',
  calendar: 'iso8601',
});
console.log(dtf.format(new Date('2024-09-09T08:00:00Z')));
```

```text
Expected:  2024 September 9, Monday at 08:00:00 AM UTC
Observed:  2024  9, Monday at 08:00:00 AM UTC          ← month "September" is missing
```

The omission is silent — no error, no warning, just a wrong string
heading downstream into logs, UIs, and serialised payloads. The
default `calendar: 'gregory'` is unaffected, which is what makes
the regression easy to miss in test suites that only exercise the
default calendar.

## Why this bug

- **Layer 2 fit.** A pure Node runtime regression in bundled ICU.
  Layer 1 (Pyodide) does not ship Node; Layer 3 (rr) is overkill
  for a deterministic single-process bug. Docker is exactly right.
- **Currently open upstream.** The issue
  ([nodejs/node#63041](https://github.com/nodejs/node/issues/63041))
  is open with `confirmed-bug` + `regression` labels and has no
  attached fix PR at the time of recipe creation. The recipe is a
  fix-detection sentinel: when upstream lands a fix in a future
  Node release, rebuilding the image flips the verdict from
  `reproduced` to `unreproduced`.
- **Deterministic.** The bug is a pure function of inputs — date,
  locale, options object — and reproduces 100 % under
  `node:26-slim`. No timing, no concurrency, no environment
  drift.
- **Cheap reproduction.** The repro is one short JavaScript
  expression evaluated under `node -e`, taking milliseconds. The
  whole image run completes in under a second.

## Verdict contract

Per [`src/layer2_docker/README.md`](../README.md):

- The gallery surfaces the **last verdict CI captured** alongside
  this page (`reproduced` / `unreproduced`, image digest, timestamp).
- The visitor's local `docker run` is the **live confirmation**.
- `reproduced` ⇔ the formatter's output does **not** contain the
  substring `"September"` for the input date `2024-09-09T08:00:00Z`
  with `calendar: 'iso8601'`.
- `unreproduced` ⇔ the month name is present, indicating the
  upstream fix has landed (or the environment has been altered to
  bypass the bug — e.g. `--icu-data-dir` pointing at a
  different ICU build).

## References

- Upstream issue: [nodejs/node#63041](https://github.com/nodejs/node/issues/63041)
  — "Missing month in `Intl.DateTimeFormat` output when using
  `calendar: 'iso8601'` in Node v24.13.0 and later".
- ECMAScript Internationalisation API spec, [`Intl.DateTimeFormat`
  → `dateStyle`](https://tc39.es/ecma402/#sec-datetimeformat-dateStyle):
  the `'full'` style is required to include the month, the day,
  and the weekday in the locale-appropriate order.
- Node.js ICU integration: [building Node.js with ICU](https://github.com/nodejs/node/blob/main/BUILDING.md#intl-ecma-402-support)
  — the bundled small-icu / full-icu split context for the
  upstream regression.
