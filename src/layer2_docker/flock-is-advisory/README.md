> [`postgres-lost-update`](../postgres-lost-update/) and
> [`bash-local-shadows-exit`](../bash-local-shadows-exit/)).
> Conforms to the catalogue conventions in

## The behaviour

Linux's `flock(2)` (and the `flock(1)` CLI on top of it) creates
**advisory** locks: only programs that *also* call `flock` see
the lock. Programs that don't — `cat`, `cp`, `grep`, every
editor's "save", every web server's log writer — ignore the lock
entirely and read or write the file freely.

A surprising fraction of developers reach for `flock` to "protect"
a file, then are surprised when arbitrary tools read it during the
hold. The `flock(2)` man page is explicit:

> A call to `flock()` may block if an incompatible lock is held by
> another process. […] **Locks created by `flock()` are
> associated with an open file description** […]. *This subsystem
> implements only advisory locks*.

## Why this bug

- **Quintessentially Layer 2.** Real Linux kernel, real
  `flock(2)` syscalls, real concurrent processes. None of this
  lives in WASM — Vivarium catalogues real OS-level surprises.
- **Deterministic verdict.** A backgrounded `flock -x file -c
  "sleep 2"` holds the lock; we verify the hold with a separate
  `flock -n -x` (which fails) and then read the file with plain
  `cat` (which succeeds). Final state is `reproduced: true` iff
  both checks land as expected.
- **Catalogue-fit.** Open / Closed status doesn't apply — this
  is documented kernel behaviour developers misread routinely.
  The page makes the surprise reproducible in seconds; the fix
  is application-side: every reader must also `flock` (or use
  `O_TMPFILE` + atomic rename, or use mandatory locks if the
  filesystem supports them, or use a database).
- **Different domain from prior pages.** First Layer 2 page is
  database isolation; second is shell scripting; this third one
  is OS / fs syscall semantics. The catalogue keeps stretching.

## Verdict contract

Per [ADR-0010](../../_context/decisions/0010-phase3-catalogue-model.md)
and [`src/layer2_docker/README.md`](../README.md):

- The gallery surfaces the **last verdict CI captured** alongside
  this page (`pass` / `fail`, image digest, timestamp).
- The visitor's local `docker run` is the **live confirmation**.
- `pass` ⇔ exclusive `flock` was held (verified by a contended
  `flock -n -x` failing) AND `cat` read the file regardless. The
  surprise reproduces.
- `fail` ⇔ kernel suddenly began enforcing `flock` as mandatory
  (it won't, but the verdict captures that branch honestly), or
  the lock was not actually held when we tested.

## Running locally — `docker run`

```bash
docker run --rm ghcr.io/aletheia-works/vivarium-flock-is-advisory:latest
```

First-pull cost: ~10 MB (`alpine:3.19` + `util-linux` extras).
Run time: ~2.5 seconds (the lock-holder sleeps 2 s).

**Verified output** (run on WSL2 kernel 6.6 / `util-linux
2.39.3`, prior to first CI run):

```
{
  "kernel_version": "6.6.114.1-microsoft-standard-WSL2",

verdict=pass — flock(2) is advisory; cat(1) bypassed the exclusive lock
```

Exit code `0`: surprise reproduced, page reports `pass`.

## References

- `flock(2)` Linux man page —
  <https://man7.org/linux/man-pages/man2/flock.2.html>.
  The "Locks created by `flock()` are associated with an open
  file description" paragraph and the "*advisory locks*"
  emphasis are the relevant lines.
- POSIX `fcntl(F_SETLK)` is similarly advisory; mandatory
  locking on Linux requires the `mand` mount option *and* the
  setgid-no-group-execute file-permission bit, and is deprecated
  in modern kernels — see <https://www.kernel.org/doc/html/latest/filesystems/mandatory-locking.html>.
