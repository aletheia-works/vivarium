## The behaviour

`xargs` splits its stdin on whitespace **by default**. A filename
like `"file with space.txt"` is therefore handed to the downstream
command as **three** arguments — `"file"`, `"with"`,
`"space.txt"` — none of which exist. The pipeline:

```bash
find . -name '*.txt' | xargs grep -l 'secret'
```

silently misses every file whose path contains whitespace. The
canonical fix uses NUL bytes as separators (a filename can
contain anything *except* `/` and `\0`):

```bash
find . -name '*.txt' -print0 | xargs -0 grep -l 'secret'
```

This is one of the most common Unix scripting gotchas, with a
direct production analogue: every CI script that processes
"all files in a directory" via the naive form silently breaks
the moment a contributor names a file with a space, an
apostrophe, or a newline.

## Why this bug

- **Quintessentially Layer 2.** Real filesystem entries with
  real path bytes, real `find` and `xargs` from GNU findutils,
  real shell pipelines. None of this lives in WASM.
- **Deterministic verdict.** A single test file with a
  whitespace name is created in `/tmp/`; the naive pipeline
  fails to find it, the `-print0`/`-0` pipeline finds it. The
  page reports `pass` iff both halves behave as documented.
- **Catalogue-fit.** Documented Unix tooling behaviour. Same
  framing as the other Layer 2 entries: Vivarium catalogues
  observable real-world surprises, not just numbered upstream
  defects.
- **Different domain again.** Postgres → bash → flock → POSIX
  tool composition. Four Layer 2 entries now span four
  different problem shapes.

## Verdict contract

Per [ADR-0010](../../_context/decisions/0010-phase3-catalogue-model.md):

- The gallery surfaces the **last verdict CI captured**.
- The visitor's local `docker run` is the **live confirmation**.
- `pass` ⇔ naive pipeline fails to find the whitespace-named
  file AND `-print0` pipeline finds it.
- `fail` ⇔ naive pipeline now finds the file (which would
  mean `xargs`'s default tokenisation changed — it won't, but
  the verdict captures that branch honestly).

## Running locally — `docker run`

```bash
docker run --rm ghcr.io/aletheia-works/vivarium-find-xargs-whitespace:latest
```

First-pull cost: ~10 MB (`alpine:3.19` + GNU findutils). Run
takes well under a second.

**Verified output** (run on Docker Desktop / WSL2, GNU
findutils 4.9.0, prior to first CI run):

```
{
  "find_version": "find (GNU findutils) 4.9.0",
  "xargs_version": "xargs (GNU findutils) 4.9.0",
  "naive_pipeline": "find … -name '*.txt' | xargs grep -l 'secret'",
  "naive_found_file": false,
  "naive_exit_code": 123,
  "nul_pipeline": "find … -name '*.txt' -print0 | xargs -0 grep -l 'secret'",
  "nul_found_file": true,
  "nul_exit_code": 0,
  "reproduced": true
}
verdict=pass — naive find|xargs splits on whitespace; -print0/-0 fixes it
```

The naive pipeline's exit 123 is `xargs`'s "one or more child
invocations exited non-zero". `grep` fails to open every
mis-tokenised path; `xargs` collects the failures and exits
with 123 to flag them.
