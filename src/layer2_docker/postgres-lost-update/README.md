# Reproduction — PostgreSQL lost update under READ COMMITTED

> Phase 3 reproduction page — first **Layer 2 (Docker)** entry in
> Vivarium's gallery. Conforms to the
> [catalogue conventions in `src/layer2_docker/README.md`](../README.md):
> a pinned `Dockerfile` + `repro.sh` + a public pre-built image,
> with the visitor reproducing the bug locally via `docker run`
> or one-click via Codespaces.

## The behaviour

The textbook **lost update anomaly**: under PostgreSQL's default
`READ COMMITTED` isolation level, two concurrent application-side
"read-then-write" transactions can silently lose one of the
writes.

```
Tx1                                  Tx2
─────────────────────────────────────────────────────────────
BEGIN
SELECT value FROM counters → 0
                                     BEGIN
                                     SELECT value FROM counters → 0
                                     UPDATE SET value = 0 + 1
                                     COMMIT
UPDATE SET value = 0 + 1
COMMIT
─────────────────────────────────────────────────────────────
Final value: 1   (expected 2 — one increment lost)
```

This is **documented PostgreSQL behaviour**, not a bug to be
fixed. Vivarium's value here is making the reproduction
**runnable in seconds** so a developer can confirm — once and
for life — that "one transaction reads, then later commits a
new value based on what it read" is unsafe under the default
isolation level. The fix is application-side: `SELECT … FOR
UPDATE`, `REPEATABLE READ` / `SERIALIZABLE` isolation, or
optimistic-concurrency check on the write.

## Why this bug

- **Quintessentially Layer 2.** A real PostgreSQL server, real
  concurrent backend processes, real MVCC. None of this lives
  in WASM — the demo is exactly what Layer 2 exists for.
- **Deterministic verdict.** With Tx1 sleeping 1 s mid-transaction
  and Tx2 sprinting through, the schedule is forced. Final
  value is `1` (lost update reproduces) or `2` (both increments
  preserved, which would mean PostgreSQL had unilaterally
  upgraded the isolation — it won't, but the verdict captures
  that branch honestly).
- **Catalogue-fit.** Open / Closed status doesn't apply — this
  is documented isolation-level behaviour developers
  *repeatedly* encounter in production. The page is a
  reproducibility guarantee, not a defect tracker entry.

## Verdict contract

Per [ADR-0010 (private memo)](../../_context/decisions/0010-phase3-catalogue-model.md)
and [`src/layer2_docker/README.md`](../README.md):

- The gallery card surfaces the **last verdict CI captured**
  alongside this page (`pass` / `fail`, image digest,
  timestamp).
- The visitor's local `docker run` is the **live confirmation**
  — paste the command, see the same exit code.
- A divergence (visitor's run disagrees with the snapshot) is
  itself a signal. Most commonly it means the visitor's
  PostgreSQL version inside the container drifted from the
  one CI built.

## Running locally — `docker run` (default tier)

```bash
docker run --rm ghcr.io/aletheia-works/vivarium-postgres-lost-update:latest
```

First-pull cost: ~80 MB (postgres:16-alpine). Subsequent pulls
hit Docker's local cache; the run takes ~10 seconds (3 s for
postgres to boot, 1 s for the staged transactions, 1 s
cleanup).

**Expected output (snippet):**

```json
{
  "postgres_version": "16.x",
  "isolation_level": "read committed (default)",
  "expected_final": 2,
  "actual_final": 1,
  "lost_update": true,
  "reproduced": true
}
verdict=pass — lost update reproduces under default READ COMMITTED isolation
```

Exit code `0`: lost update reproduced, page reports `pass`.

## Why the schedule is reliable

The reproduction depends on Tx2 committing while Tx1 is still
sleeping. We force the schedule by:

1. Starting Tx1 in a background subshell, then sleeping 0.3 s
   in the foreground so Tx1 has time to issue its `SELECT` and
   enter `sleep 1`.
2. Tx2 (foreground) does its `SELECT` and `UPDATE; COMMIT;`
   immediately after, well within Tx1's 1-second sleep window.
3. Tx1 wakes up after `sleep 1`, by which time Tx2 has
   committed; Tx1's `UPDATE` overwrites Tx2's value.

The only realistic failure mode is the host being so loaded
that Tx2 takes more than 700 ms to commit. On a vanilla
`docker run` even on a cold laptop this is comfortably wide;
on shared CI runners it may need adjustment. If the verdict
ever flips to `fail` because of timing rather than a real
isolation change, we widen the sleep budget.
