## The behaviour

In bash, the `local` builtin completes successfully (exit 0) as
soon as it has created the local variable. When `local` is
combined with command substitution, **`local` itself is the most
recent command** on that line, so `local`'s 0 exit code
**shadows** the exit code of whatever ran on the right-hand
side. `set -e` therefore sees a successful command and does not
trigger, even though the RHS just failed:

```bash
# Without `local` ŌĆö set -e correctly catches the failure.
set -e
x=$(false)
echo "unreachable"   # ŌåÉ not printed, set -e killed us at the
                     #   failing assignment

# With `local` ŌĆö set -e is silently bypassed.
demo() {
  set -e
  local x=$(false)
  echo "this DID print"  # ŌåÉ printed, set -e never fired
}
```

This is **documented bash behaviour**, not an upstream defect to
be fixed. Bash maintainers and the POSIX standards trail
explicitly state that the assignment statement's exit code is
the exit code of the *last command run*, and `local`'s own
expansion + variable-creation step counts as that last command.
The fix is application-side: separate the `local` declaration
from the command substitution.

```bash
# Idiomatic fix.
local x
x=$(failing_cmd) || handle_failure
```

## Why this bug

- **Quintessentially Layer 2.** Real bash, real subshells, real
  POSIX-shell semantics. None of this lives in WASM ŌĆö Vivarium
  catalogues real toolchain behaviour developers stumble into in
  production.
- **Deterministic verdict.** Two subshells run the demo with and
  without `local`. The bare-assignment subshell exits non-zero
  (set -e kills it); the `local`-assignment subshell exits 0
  (set -e bypassed). Final verdict is `reproduced` iff both expected
  exit codes match.
- **Catalogue-fit.** Open / Closed status doesn't apply ŌĆö bash's
  behaviour is documented and accepted upstream. The page's
  value is making the reproduction **runnable in seconds** so a
  developer can confirm the surprise once and for life.
- **Different domain from `postgres-lost-update`.** First Layer 2
  page is database isolation; this one is shell scripting.
  Together they show the catalogue model generalises beyond a
  single problem domain.

## Verdict contract

Per [`src/layer2_docker/README.md`](../README.md):

- The gallery surfaces the **last verdict CI captured** alongside
  this page (`reproduced` / `unreproduced`, image digest, timestamp).
- The visitor's local `docker run` is the **live confirmation**.
- `reproduced` Ōćö bare-assignment subshell exits non-zero AND
  `local`-assignment subshell exits 0 (the surprise reproduces).
  `unreproduced` Ōćö bash now propagates the RHS exit code through
  `local`, or both subshells behaved the same way (which would
  imply a deeper change in bash's evaluation model).

## References

- bash manual, "SHELL BUILTIN COMMANDS ŌåÆ local". The exit code
  paragraph notes that `local`'s status is its own success, not
  the RHS's.
- POSIX 2017, Section 2.9.1: "Simple Commands" ŌĆö exit status
  of an assignment-only command is the exit status of the *last*
  command substitution performed, *unless* the assignment is
  performed by a builtin (`local`, `declare`, `export`,
  `readonly`).
- Greg's wiki, [BashFAQ #105](https://mywiki.wooledge.org/BashFAQ/105)
  ŌĆö `local` in bash: gotchas and idioms.
