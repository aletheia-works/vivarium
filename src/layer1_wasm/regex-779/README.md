# Reproduction — rust-lang/regex#779

> Phase 2 reproduction page — third non-Pyodide entry in Vivarium's
> gallery (after [`ruby-21709/`](../ruby-21709/) and
> [`php-12167/`](../php-12167/)) and the first to ship a **compiled**
> Rust artefact rather than a CDN-hosted single-runtime bundle.
> Conforms to `vivarium-contract: v1`.

## The bug

[rust-lang/regex#779](https://github.com/rust-lang/regex/issues/779)
— Regex algebra requires `(re)+` to be equivalent to `(re)(re)*`.
With the multiline pattern over a small haystack the equivalence
breaks:

```rust
let haystack = "a\naaa\n";
let re_plus     = Regex::new("(?m)(^|a)+").unwrap();
let re_expanded = Regex::new("(?m)(^|a)(^|a)*").unwrap();

let matches_plus     = re_plus.find_iter(haystack)
    .map(|m| (m.start(), m.end())).collect::<Vec<_>>();
let matches_expanded = re_expanded.find_iter(haystack)
    .map(|m| (m.start(), m.end())).collect::<Vec<_>>();

// Reported (regex 1.8.4 — pinned in this repro's Cargo.toml):
//   matches_plus     == [(0, 0), (2, 2), (3, 5), (6, 6)]
//   matches_expanded == [(0, 1), (2, 5), (6, 6)]
```

The two match lists should be identical. They're not. The same bug
reproduces in RE2 and Go's regexp; PCRE2 gets it right, so this is
an algebraic regex-engine issue, not a language-spec ambiguity.
Open upstream at the time of writing.

## Why this bug

- Pure Rust `regex` crate — no FFI, no native extensions, no I/O.
- Verdict reduces to a boolean — `matches_plus != matches_expanded`
  — so the page emits a mechanically-distinguishable `pass` /
  `fail`.
- Open upstream (Status: Open). When a fix lands the page flips to
  `fail` and becomes a fix-detection signal.
- Algorithmic / data-structure bug, not LLVM-target-specific —
  reproduces equally on `wasm32-wasip1`, `x86_64-unknown-linux-gnu`,
  etc.
- Demonstrates that Vivarium's Layer 1 contract generalises to
  **compiled** runtimes, not just interpreters with a CDN-hosted
  bundle. The deploy workflow runs `cargo build` once on push and
  serves the resulting `.wasm` next to `index.html`.

## Verdict contract — `vivarium-contract: v1`

The page conforms to the contract canonicalised in
[`../_shared/verdict.ts`](../_shared/verdict.ts). The `result` field
of the envelope reports `pattern_plus`, `pattern_expanded`,
`matches_plus`, `matches_expanded`, `reproduced`, and `exit_code` —
enough for downstream tooling to distinguish the specific shape of
any future change.

A `pass` means **the bug reproduced** — the two patterns produced
different match lists *and* the wasm exited 0. A `fail` means either
the runtime ships a fix (lists agree, exit 1), the wasm errored
before producing a result, or the WASI shim could not load.

## Native verification — same reproduction under a real Rust

The `Cargo.toml` and `src/main.rs` work as a normal Rust binary. The
`.mise.toml` at the repo root pins Rust to a stable toolchain
matching CI:

```bash
mise install                                           # one-time per machine
cd src/layer1_wasm/regex-779
mise exec rust -- cargo run --release
# verdict=pass — `(re)+` and `(re)(re)*` disagree on this haystack
```

`cargo run` uses the host target by default, so this exercises the
exact same logic against the regex crate built for your platform —
not a wasm artefact. CI also runs `cargo build --release --target
wasm32-wasip1` from this directory and copies the resulting `.wasm`
into the Pages artefact.

## CI build of the wasm artefact

`deploy-docs.yml` adds, for every directory matching the
`<slug>/Cargo.toml` shape:

```bash
cd src/layer1_wasm/<slug>
mise exec rust -- cargo build --release --target wasm32-wasip1
cp target/wasm32-wasip1/release/repro.wasm <slug>/repro.wasm
```

The first run hydrates `~/.cargo/registry` with `regex` and
`serde_json` (~10 MiB). Subsequent CI runs hit the registry cache.
Wall-clock cost is ~30–60 s depending on cache state.

## Running locally — in-browser

```bash
# 1. Build the Rust artefact once (or re-build after src/ changes).
cd src/layer1_wasm/regex-779
mise install                              # picks up rust + bun
mise exec rust -- cargo build --release --target wasm32-wasip1
cp target/wasm32-wasip1/release/repro.wasm ./repro.wasm

# 2. Build the page TypeScript.
cd ..
mise exec bun -- bun install
mise exec bun -- bun run build

# 3. Serve.
python -m http.server -d . 8767
# open http://localhost:8767/regex-779/
```
